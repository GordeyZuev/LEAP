# Frontend: дизайн, стек и обоснование

Документ фиксирует **планы и договорённости** по фронтенду в монорепозитории LEAP (корень репозитория: Docker Compose + документация; **`backend/`** — FastAPI/Celery; **`frontend/`** — будущий UI). Код приложения здесь пока не обязателен; это ориентир для следующего этапа.

---

## 1. Монорепозиторий: зачем так

- **Один репозиторий** упрощает согласование версий API и UI, общие типы (при желании — генерация клиента из OpenAPI), и релизы.
- **Корень** держит `docker-compose.yml` и инфраструктурные вещи; **не смешиваем** Python-цели и Node-цели в одном Makefile — у бэкенда свой `backend/Makefile`, у фронта будут свои `package.json` / `pnpm` scripts.
- Бэкенд уже отдаёт OpenAPI (`/openapi.json`) — это естественная точка контракта для фронта.

---

## 2. Рекомендуемый стек

| Слой | Выбор | Почему |
|------|--------|--------|
| Фреймворк | **Next.js** (App Router), **TypeScript** | SSR/SSG для лендинга и SEO, маршрутизация, предсказуемая интеграция с хостингом (Vercel/Node/Docker). |
| UI-компоненты | **shadcn/ui** (Radix + Tailwind) | Доступные примитивы, владение кодом компонентов в репозитории (не «чёрный ящик» библиотеки), быстрый консистентный UI под админку/дашборд. |
| Стили | **Tailwind CSS** | Тот же экосистемный стандарт, что у shadcn; удобно для дизайн-токенов и адаптива. |
| Данные с API | **TanStack Query (React Query)** | Кэш, повторы, инвалидация после мутаций — критично для «записей, задач, статусов» LEAP. |
| Формы и валидация | **React Hook Form** + **Zod** | Совместимо с shadcn Form; схемы можно шарить с типами ответов API. |
| HTTP-клиент | **fetch** или **ky** / обёртка под OpenAPI | Минимум зависимостей; при росте — codegen клиента из OpenAPI. |

Альтернативы (кратко): **Vite + React** — если не нужен SSR/SEO из коробки; **Remix** — если приоритет формам и веб-стандартам. Для продукта с публичным лендингом и кабинетом **Next.js** остаётся самым прямым компромиссом.

---

## 3. Два «лица» продукта: лендинг и приложение

Обсуждалось разделение **маркетингового сайта** и **авторизованного приложения**. Варианты:

1. **Одно Next.js-приложение**, разделение через **route groups**: `(marketing)` и `(app)` — общий дизайн-токен, один деплой, проще CI.
2. **Два приложения** в `frontend/` (например `apps/web` и `apps/app` в будущем pnpm workspace) — если команды/релизы должны быть независимы или лендинг статически выкладывается отдельно.

Рекомендация на старт: **один Next-проект с route groups**, пока нет жёсткого требования к независимым релизам.

---

## 4. Дизайн и UX-принципы

- **Плотный, рабочий интерфейс** для кабинета (таблицы, фильтры, статусы пайплайна записей) — shadcn Data Table + чёткая типографика.
- **Лендинг** — отдельный визуальный тон (больше воздуха, иллюстрации), но **те же токены** (цвета, радиусы, шрифты), чтобы не ощущаться «другим продуктом».
- **Состояния загрузки и ошибок** явно (скелетоны, toast от shadcn/Sonner), особенно для долгих операций (транскодинг, загрузки).

---

## 5. Интеграция с `backend/`

- Базовый URL API: переменная окружения (например `NEXT_PUBLIC_API_URL` для браузера; для серверных компонентов/прокси — отдельный внутренний URL при Docker).
- **CORS** на бэкенде уже настраивается через `settings`; в dev обычно `http://localhost:3000`.
- Контракт: **OpenAPI** → по мере стабилизации можно подключить **openapi-typescript** или **orval** для типобезопасных клиентов.

---

## 6. Аутентификация

- Бэкенд выдаёт **JWT**; на фронте типичная схема: хранение refresh (httpOnly cookie — предпочтительнее с точки зрения XSS) или согласованный с бэкендом поток (уже есть OAuth-платформы в API).
- **Не дублировать** бизнес-логику авторизации на фронте: роли и права по возможности отражать данными с API (и кэшировать в Query).

---

## 7. Локальная разработка

- Пакетный менеджер: **pnpm** (уже заложено в корневом `.gitignore` под `frontend/`).
- Типичный поток: поднять Postgres/Redis через корневой `make docker-up`, API — `cd backend && make api`, фронт — `cd frontend && pnpm dev` (после инициализации проекта).

---

## 8. Деплой (на будущее)

- **Vercel** — естественно для Next.js (лендинг + app).
- **Docker** — один образ Next (Node) или статический экспорт лендинга + отдельный хостинг app — по требованиям инфраструктуры.
- Секреты фронта только для публичных ключей; **JWT secret и encryption** остаются на бэкенде.

---

## 9. Следующие шаги (чеклист)

1. Инициализировать Next.js + TypeScript в `frontend/` (и зафиксировать версии в lockfile).
2. Подключить Tailwind и shadcn по официальной инструкции.
3. Завести `frontend/.env.example` с `NEXT_PUBLIC_API_URL`.
4. Первый экран: health/login или список записей против staging API.
5. При необходимости — вынести общие типы или codegen из `backend` OpenAPI.
